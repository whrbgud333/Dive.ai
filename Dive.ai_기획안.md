Dive.ai: 시나리오, 캐릭터 빌더 및 인터랙티브 캐릭터 채팅 웹서비스

  1. 프로젝트 개요
   * 프로젝트명: Dive.ai(가칭)
   * 목표: AI 허브의 문화콘텐츠 스토리 데이터, 동아시아 고전 스토리 데이터, 웹소설 데이터 등을 활용하여 사용자의 단편적인 소재를 완성도 높은 시나리오와 캐릭터로 제작하고, 이를 분기형
     인터랙티브 구조로 변환하여 AI 캐릭터와 소통하며 자신만의 엔딩을 만들어가는 웹 서비스 개발

  기획 배경
   * AI 캐릭터 채팅 시장의 급성장: 전 세계적으로 AI 캐릭터와 대화하는 서비스가 폭발적으로 성장하고 있으나, 대다수 서비스가
     '단순 일상 대화'에 그쳐 사용자가 금방 지루함을 느끼는 '콘텐츠의 휘발성' 문제가 있으며 사용자가 직접 대화를 이끌어 나가야 한다는 부담감이 있음.

   * 타인 의존적 콘텐츠 소비 구조: 현재의 AI 캐릭터 채팅 서비스는 주로 타인이 이미 만들어 놓은 페르소나나 세계관에 유저가 '손님'으로 참여하는 형태에 치중되어 있음. 이러한 형태보다는 사용자가 직접 시나리오와 캐릭터를 만들고, 본인이 직접 만든 시나리오 속에서 캐릭터와 소통하고, 이야기를 진전시키는 데 중점을 둔 서비스를 만들려고 함. 사용자는 단순한 서비스 이용자가 아닌 '서사의 주인공이자 창작자'로서 본인이 직접 창조한 이야기의 맥락 속에서 캐릭터와 대화를 통해 몰입감을 극대화 할 수 있음.

  문제 해결 방향
   * [대화에서 '플레이'로] 그냥 대화가 아닌, 시나리오 빌더를 통해 '엔딩'이라는 명확한 목표를 가진 인터랙티브 서사 구조를 구축하여 몰입감과 재방문 동기를 확보함.
   * [선형 서사의 입체화] 사용자의 짧은 소재를 LLM이 기승전결로 확장하고, 이를 다시 분기형 인터랙티브 구조로 변환하여 유저가 대화를 직접 이끌어 나가지 않아도 유저의 선택에 따라 결과가 달라지는 입체적인 인터랙티브 서사를 구축함.
   * [상황 인식형 멀티모달] 분기점에서 선택을 하면 분기별 대화 맥락과 캐릭터의 감정 상태를 분석하여 실시간으로 생성되는 AI 이미지를 통해 시각적 서사 경험을 완성함.

  2. 핵심 기능

  AI 시나리오, 캐릭터 빌더 (Scenario & Character Builder)
   * 기능: 사용자가 입력한 키워드/소재를 바탕으로 기승전결이 갖춰진 초기 시나리오 생성, 주인공 캐릭터 생성
   * 데이터: 문화콘텐츠 스토리 데이터, 동아시아 고전 스토리 데이터, 직접 수집한 웹소설 데이터
   * 기술: LLM + RAG. 장르별 흥행 서사 패턴을 반영하여 복선과 갈등 구조가 설계된 트리트먼트 작성

  AI 시나리오 트랜스포머 & 챗 (Scenario Transformer & Chat)
   * 기능: 선형 시나리오를 인터랙티브 분기점(Choice Points)으로 변환하고, AI 캐릭터 채팅 및 분기별 상황에 맞는 이미지 생성
   * 기술:
       * LLM을 활용한 Narrative Branching: 흥행 콘텐츠의 갈등 구조를 학습하여 몰입감 있는 선택지 생성
       * API(Leonardo.ai, Civitai): 분기별 상황에 맞는 이미지 생성

  로어북(Lorebook)
  * 기능: 사용자가 직접 세계관의 핵심 설정(지명, 인물 관계, 고유 명사, 역사적 사건 등)을 등록 및 관리하며, AI가 대화 중 이를 실시간으로 참조하여 서사의 일관성을 유지
  * 기술:
      * 이벤트 스트리밍 및 자동 인덱싱: 시나리오 생성 시 주요 사건을 자동으로 추출하여 벡터 DB에 색인화함.
      * 시맨틱 검색(Semantic Search) : 단어의 단순 일치 여부가 아닌, 문맥적 의미의 유사성을 분석하여 관련 기억을 정교하게 호출함.
      * 실시간 컨텍스트 주입 (Real-time Context Injection): 대화 흐름을 분석하여 필요한 로어북 데이터를 시스템 프롬프트에 동적으로 결합.
      * 서사 일관성 엔진 (Narrative Consistency Engine): 과거의 사건과 현재의 대화가 충돌하지 않도록 AI의 기억 우선순위를 관리.

로어북 워크플로우
1. 사건 저장 (Automatic Event Extraction)
사용자가 시나리오 초안을 확정하는 순간, **'시나리오 빌더'**가 전체 줄거리를 분석. LLM이 "이 줄거리에서 나중에 기억해야 할 핵심 사건은 무엇인가?"를 판단하여 [주체, 객체, 행동, 시간, 결과] 단위로 쪼개어 데이터베이스(Vector DB)에 저장.

결과: "어제/주인공/악당과 싸움/주인공 승리"라는 데이터가 로어북의 '사건 로그' 섹션에 자동으로 기록.

2. 트리거 (Semantic Search & Retrieval)
사용자가 채팅창에 대화를 입력하면, 시스템은 그 문장을 서버로 보내기 전 실시간으로 데이터베이스를 검색.

단순히 '단어'가 일치해야 하는 것이 아니라, 의미가 유사하면 소환되는 **'시맨틱 검색(Semantic Search)'**을 사용. 예를 들어 사용자가 '악당' 대신 '그 녀석'이라고 말해도, 문맥상 '악당'을 지칭한다고 판단되면 관련 기억을 끌어옴.

3. 반영 (Context Injection)
검색된 기억 조각들을 AI에게 보내는 프롬프트(지시문)의 **'숨겨진 영역(System Prompt)'**에 슬쩍 끼워 넣음.

실제 작동 예시 (유저에겐 안 보임):

[시스템 지시문] (자동 삽입됨)
※ 현재 맥락 데이터: 어제 주인공은 악당과 치열하게 싸워 승리했습니다. 악당은 현재 부상을 입고 도망친 상태입니다.
[유저 입력] "그 녀석, 지금 어디쯤 갔을까?"
[AI 답변] "어제 그렇게 당하고도 정신을 못 차렸나 보군. 부상이 깊으니 멀리는 못 갔을 거야."

  3. 서비스 워크플로우 (Workflow)

  사용자가 접속하여 엔딩에 이르기까지의 5단계 과정입니다.

   1. [Input: 소재 입력]
       * 사용자가 선호 장르, 핵심 키워드, 대략적인 소재를 입력합니다. (예: "조선시대 배경의 퇴마사 이야기", "배신당한
         복수극")
   2. [Build: 시나리오 초안, 캐릭터 생성]
       * Scenario Builder가 유저가 입력한 소재와 데이터를 참조하여 기승전결이 뚜렷한 줄거리를 작성하고, 주요 인물들의 페르소나를 정의합니다. 그 후에 Scenario Builder를 통해 대화할 주인공 캐릭터를 생성하고 유저가 참여할 시나리오 속 인물도 생성합니다.
   3. [Transform: 인터랙티브 설계]
       * Scenario Transformer가 줄거리 내 핵심 갈등 지점을 분석하여 유저가 개입할 수 있는 '분기점'을 생성합니다. 이때
         장르 특화된 '복선'이나 '반전' 요소가 선택지에 배치됩니다.
   4. [Play: 실시간 캐릭터 채팅]
       * 사용자가 시나리오 속 인물이 되어 AI 캐릭터와 대화합니다. AI는 학습한 데이터를 기반으로 깊이 있는
         말투와 가치관을 구사하며 유저의 선택에 반응합니다.
   5. [Visualize & Ending: 상황별 시각화 및 엔딩]
       * 채팅 중 주요 사건이 발생할 때마다 해당 상황에 최적화된 AI 캐릭터 이미지가 생성되어 몰입감을 높입니다. 유저의
         선택에 따라 최종적인 멀티 엔딩 중 하나에 도달합니다.

  4. 핵심 강점
  * "몰입도 극대화": 타인이 설계한 세계관에 손님으로 참여하는 기존 방식에서 탈피하여, 유저가 직접 창조한 서사 속의 '주인공'이 되는 경험을 할 수 있음. 본인이 만든 캐릭터와 서사 속에서 이야기를 진전시킴으로써 몰입도를 극대화 할 수 있음.
   * "분기형 인터랙티브": 단순한 감정 교류나 일상적 대화를 넘어, 대화 자체가 플롯을 전개하는 실질적인 동력이 됨. 캐릭터와의 대화 내용에 따라 사건의 전개가 변하고 갈등이 해결되는 분기별 인터랙션을 통해, 유저는 자신의 선택이 세계관에 실질적인 영향을 미치고 있음을 체감하고 이는 몰입도 상승으로 이어짐.
   * "완결성 있는 경험": 끝이 없는 채팅보다 '엔딩'이라는 보상을 제공함으로써 유저에게 한 편의 이야기를 직접 완성했다는 성취감을 주고, 다른 엔딩을 보기 위한 재방문 동기를 얻게 됨.

7. 개발 파이프라인 (Development Pipeline)

Phase 1: 데이터 전처리 및 지식 베이스 구축 (Data & Embedding)
* 데이터 정제: AI 허브에서 수집한 데이터를 문단 단위로 청킹(Chunking)하고 클리닝.
* 벡터 임베딩: LangChain과 OpenAI Embedding 모델 등을 활용하여 정제된 데이터를 벡터화.
* Vector DB 구축: ChromaDB 또는 Pinecone을 사용하여 시맨틱 검색이 가능한 고성능 지식 베이스를 구축.

Phase 2: 시나리오 & 로어북 엔진 개발 (Core Engine)
* RAG 파이프라인 설계: 사용자 입력 키워드에 따라 관련 서사 모티프를 소환하는 Retrieval-Augmented Generation 로직을 구현.
* 서사 분기 로직(Branching Logic): LLM이 생성한 선형 스토리를 분석하여 핵심 갈등 지점에서 선택지를 생성하는 프롬프트 체인을 설계.
* 로어북 자동 업데이트 시스템: 대화 중 발생하는 사건을 추출하여 Vector DB에 실시간으로 반영하는 인덱싱 자동화 기능을 개발.

Phase 3: 캐릭터 페르소나 및 채팅 최적화 (Dialogue System)
* 최적화 프롬프팅: 서사에 맞게 캐릭터의 페르소나와 말투를 파인튜닝(Fine-tuning)하거나 정교한 시스템 프롬프트를 설계.
* 시맨틱 검색 엔진 고도화: 대명사 처리 및 문맥 이해도를 높이기 위해 코사인 유사도 기반의 검색 알고리즘을 최적화.

Phase 4: 멀티모달 연동 및 백엔드 구축 (Integration)
* 이미지 생성 파이프라인: Leonardo.ai 또는 Civitai API를 연동하여 대화 맥락에 맞는 캐릭터 이미지를 생성하는 모듈을 구축.
* 백엔드 API 개발: **Streamlit**이나 **Gradio**를 사용하여 시나리오 생성, 채팅, 이미지 소환 기능을 통합하는 API 서버를 개발.

Phase 5: 서사 일관성 테스트 및 배포 (QA & Deployment)
* Consistency Test: 장기 대화 시 캐릭터 붕괴나 설정 충돌이 일어나는지 할루시네이션(환각) 테스트를 진행.
* UI/UX 최적화: 인터랙티브 선택지 중심의 웹 인터페이스를 구축하고 최종 서비스를 배포.
